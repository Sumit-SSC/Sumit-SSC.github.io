Project Overview

Fraud Intelligence & Risk Analytics is a SQL-first fraud analytics platform that mirrors how payments and card programs run fraud operations in production. It generates a realistic, multi-table transaction environment (>1.8M rows), builds a layered SQL pipeline (raw → staging → features), exports analytics tables to Power BI, and provides a multi-page Streamlit investigation console for fraud analysts.

Instead of only training a single classification model, the project focuses on the full decision system: how transactions become risk signals, how those signals feed queues and dashboards, and how investigators consume them day-to-day.

Business Context & Problem Framing

Fraud teams in banks, card schemes, and fintechs face two opposing failure modes:

  • Too many alerts: analysts drown in false positives, queue SLAs slip, and costs escalate.
  • Too few alerts: fraud slips through undetected, driving losses and regulatory exposure.

Raw transaction data is noisy: the same card and device can be safe or risky depending on velocity, geography, merchant behavior, and customer history. A useful fraud analytics system needs to:

  • Transform raw events into business-readable features (velocity, merchant risk, patterns).
  • Expose risk scores and explanations that investigators can trust.
  • Provide dashboards and queues that prioritize the highest-value work.

Goal

Build an end-to-end fraud analytics prototype that a fraud operations lead could recognize: data model, SQL layers, BI dashboards, and investigation screens—not just a Jupyter notebook.

Architecture & Analytical Approach

1. Data Model & Pipeline Stages

The project is structured as a modern analytics pipeline: 

Raw Data → SQL Staging → Feature Tables → BI Exports → Apps (Power BI, Streamlit)

The sql/ folder implements a canonical layered design:

  • DDL: card, device, merchant, transaction, rule, and investigation tables.
  • Staging: clean & standardize source fields, derive canonical IDs, normalize currencies.
  • Features: aggregation tables for velocity, merchant concentration, behavioral deltas, etc.
-- Example: velocity feature engineering (simplified)
WITH tx_window AS (
  SELECT
    card_id,
    tx_id,
    tx_timestamp,
    amount,
    COUNT(*) OVER (
      PARTITION BY card_id
      ORDER BY tx_timestamp
      RANGE BETWEEN INTERVAL 15 MINUTE PRECEDING AND CURRENT ROW
    ) AS tx_count_15m,
    SUM(amount) OVER (
      PARTITION BY card_id
      ORDER BY tx_timestamp
      RANGE BETWEEN INTERVAL 15 MINUTE PRECEDING AND CURRENT ROW
    ) AS tx_amount_15m
  FROM raw.transactions
)
SELECT
  card_id,
  tx_id,
  tx_timestamp,
  tx_count_15m,
  tx_amount_15m,
  tx_count_15m >= 5      AS high_velocity_flag,
  tx_amount_15m >= 5000  AS high_velocity_amount_flag
FROM tx_window;

├── streamlit_app.py           # Unified entry point (mode selector)
├── streamlit_app/             # Routing and shared config

├── app-vizulation/            # Lightweight analytics app
├── sql/                       # DDL, staging, features

├── notebooks/                 # Storytelling + experimentation
├── data/                      # raw / processed / BI-ready
├── models/                    # persisted ML models
└── powerbi/ & docs/          # BI artifacts and documentation

The typical run sequence on a new machine: 

# Install environment (uv recommended)
uv sync

# Generate synthetic but realistic payments data
uv run python src/generate_raw_data.py

# Load data into MySQL (or chosen warehouse)
uv run python src/load_raw_data.py

# Execute SQL pipeline (DDL, staging, features)
# See sql/README.md for exact order

# Export BI-ready tables
uv run python src/export_bi_data.py

# Launch Streamlit investigation console
uv run streamlit run streamlit_app.py

3. BI Dashboards & Investigation Apps

Two consumption layers sit on top of the analytics tables:

  • Power BI Fraud Analytics Dashboard – high-level KPIs (fraud rate, loss, false-positive rate), trends, and breakdowns by merchant, MCC, geography, and channel.
  • Streamlit Investigation Console – multi-page app for analysts: transaction overviews, risk explanations, analytics views, and exportable documentation.

Insights, Findings & Business Impact

4. Key Findings

  • Velocity is a strong risk indicator. Cards with unusual short-window velocity (e.g., ≥5 transactions in 15 minutes or large cumulative amounts) show materially higher fraud rates in synthetic data, echoing real-world patterns in card and wallet fraud.
  • Merchant risk is concentrated. A small set of merchants/MCCs contributes a disproportionate share of flagged and confirmed fraud, suggesting targeted monitoring and contractual controls.
  • Behavioral anomalies matter. Deviations from a card’s historical behavior (spend spikes, geography shifts) provide high-signal features, especially when combined with basic rules.
  • Queue design is as important as scoring. Risk scores are surfaced as prioritized queues, with explanations and filters, instead of a flat list of alerts.

5. Trade-offs & Failure Modes

The project deliberately focuses on analytics and decision support rather than only optimizing a single ML model. Some trade-offs:

  • Synthetic data vs real-world noise. Synthetic data preserves privacy but cannot capture all long-tail patterns and adversarial behavior; the focus is on structure and workflow.
  • Limited ML complexity. Models (where used) are intentionally simple and interpretable; the heavy lifting is in SQL feature engineering and operationalization.
  • Single-tenant setup. The current design assumes one dataset and one organization; multi-tenant, multi-market architectures would add complexity around schemas and metadata.

6. Business Impact & Relevance

For interviews or stakeholder reviews, the platform demonstrates:

  • Ability to design a full fraud analytics system, not just a model notebook.
  • Familiarity with risk operations metrics (false-positive rate, queue SLAs, investigation efficiency).
  • Comfort with SQL pipelines, BI exports, and building analyst-facing tools in Streamlit.

7. Limitations & Future Enhancements

  • Real-time streaming. Current flows are batch-oriented; adding Kafka or Kinesis-style streaming would mirror real-time authorization flows.
  • Model monitoring. The platform could track model drift, threshold stability, and population stability indices for fraud models.
  • Case management integration. Integrating with ticketing/case systems would complete the loop from alert → investigation → resolution → feedback.
