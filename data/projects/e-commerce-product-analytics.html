Project Overview

E-commerce Product Analytics is an end‑to‑end analytics stack that turns synthetic clickstream and order data into decision‑ready funnels, cohort retention, and A/B test readouts, using DuckDB + SQL at the core and BI‑ready exports for storytelling.

Executive Summary

  • Business problem: product and growth teams need trustworthy funnel and retention metrics and a clean way to read experiments, not just one‑off charts or ad‑hoc SQL.
  • What was built: a realistic e‑commerce dataset (~120k users, ~1M events, ~45k orders), a DuckDB‑based SQL pipeline, notebooks for stats, and exports for Power BI dashboards.
  • Why it matters operationally: gives teams a reproducible way to see where users drop off, how cohorts behave, and whether experiments should ship.
  • Key insights: the largest synthetic drop‑off is between product view and add‑to‑cart, early cohorts retain better, and A/B tests become auditable when the full path is in code.
  • Indicative impact: a stack like this shortens the path from raw events to “ship / hold” decisions and reduces the risk of mis‑interpreting funnel lifts or retention changes.

Business Problem: Funnels, Retention & Experiments at Scale

E‑commerce teams repeatedly ask the same questions:

  • Where are users dropping in the funnel? (visit → view → add‑to‑cart → checkout → purchase)
  • How do different cohorts retain? (by signup week, device, channel, or campaign)
  • Did this experiment actually work? (and is the effect size big enough to ship?)

Many analyses answer these questions once, in a notebook, and then get lost. Real teams need a repeatable analytics pipeline that can be re‑run, audited, and plugged into BI tools—not just screenshots.

  • Generate or ingest realistic clickstream + order data.
  • Materialize consistent, queryable tables for funnels and cohorts.
  • Support A/B test readouts with statistically sound methods.
  • Export tidy tables so Power BI and notebooks tell the same story.

Architecture, Methodology & System Design

1. How the analytics stack is built

The project is designed to look like a real e‑commerce analytics stack, not a single notebook. Everything from data generation to BI exports is expressed as code, so funnels, retention tables, and experiment metrics can be recreated on demand.

DuckDB acts as a local analytics engine, with Python handling generation and loading, while SQL defines the core metrics that BI and notebooks consume. This keeps the system lightweight but production‑shaped. DuckDB was chosen for its SQL compatibility (works with standard SQL queries), performance on analytical workloads, and ease of setup (no server required). In production, teams might use BigQuery, Snowflake, or Redshift, but the SQL patterns remain the same.

Code‑first philosophy: Every metric (funnel conversion, cohort retention, A/B lift) is defined in SQL or Python, not in a BI tool’s UI. This makes metrics auditable, version‑controlled, and reproducible. When stakeholders ask “how did you compute retention?”, the answer is “run this SQL script” rather than “I clicked these buttons in Power BI”.

Separation of concerns: Data generation and loading live in Python; SQL handles transformations and metrics; notebooks validate results with statistics; BI consumes pre‑computed tables. This separation makes it easy to swap components (e.g., replace DuckDB with BigQuery) without rewriting everything.

2. Data Landscape & Event Model

The dataset simulates a mid‑sized e‑commerce site with a clear event model: each user generates events (page view, product view, add‑to‑cart, checkout, purchase) with timestamps and optional properties (product id, session id). Orders are linked to users and events so that funnels and cohorts can be computed consistently.

Event taxonomy: Events follow a standard e‑commerce taxonomy: page_view (any page load), product_view (viewing a product detail page), add_to_cart (adding a product to cart), checkout_start (initiating checkout), and purchase (completing an order). Each event includes a user ID, timestamp, session ID (derived from sessionization), and optional properties (product ID, category, price, device type, channel). This structure mirrors how real analytics platforms (e.g., Google Analytics, Amplitude) model e‑commerce events.

  • ~120,000 users with realistic signup dates, device types, and acquisition channels (organic, paid, referral).
  • ~1,000,000 events (page views, product views, add‑to‑cart, checkout, purchase) spanning a 12‑month period with seasonality (holiday spikes, weekend patterns).
  • ~45,000 orders with product details, quantities, prices, and order totals. Orders are linked to users and events so that purchase funnels and revenue metrics can be computed.
  • ~400,000 sessions (after 30‑minute timeout sessionization). Sessions group events into logical user journeys, enabling funnel analysis and session‑level metrics (bounce rate, pages per session, session duration).

Data generation strategy: The synthetic data is generated to match realistic e‑commerce patterns: conversion rates (visit → purchase ~2–3%), retention curves (week‑1 retention ~40%, week‑4 retention ~20%), and seasonality (holiday spikes, weekend patterns). This ensures that funnels, cohorts, and A/B tests behave plausibly, making the pipeline credible for real‑world use cases.

3. Metrics, KPIs & North Star

The stack is built around a small set of product analytics KPIs. The north star is funnel conversion: the share of sessions that reach each step (visit → product view → add‑to‑cart → checkout → purchase). Secondary KPIs include cohort retention (e.g., week‑1 retention by signup week), A/B test lift and statistical significance, and session‑level metrics (bounce rate, pages per session). These are exposed in Power BI and in notebooks so that product and growth teams can answer “where do we drop off?” and “did this experiment work?” in one place.

4. Funnel & Cohort Logic

Funnels and cohorts are implemented as a sequence of SQL scripts in sql/analytics/. Sessionization (30‑minute timeout) groups events into sessions, and derived tables flag funnel steps and cohort membership for retention analysis.

Sessionization logic: Events are grouped into sessions using a 30‑minute inactivity window. If a user has no events for 30 minutes, the next event starts a new session. This mirrors how many analytics platforms (e.g., Google Analytics, Amplitude) define sessions, balancing granularity with business relevance.

Funnel steps: Each session is flagged for funnel progression: visit (any page view), product_view, add_to_cart, checkout_start, purchase. A session can have multiple product views or add‑to‑cart events, but the funnel logic captures the first occurrence of each step so conversion rates are consistent.

Cohort assignment: Users are assigned to cohorts based on their first signup or first purchase date. Retention is then measured as the share of users in a cohort who return (e.g., make a purchase) in subsequent periods. This allows product teams to compare how different cohorts (e.g., signup week, acquisition channel) retain over time.

Cohort Retention Matrix (SQL)

This SQL transform computes the user cohort retention matrix by determining first purchase dates and tracing month-by-month repeat orders: 

-- Step 1: Find user acquisition cohort (first purchase month)
WITH user_signup AS (
  SELECT 
    user_id,
    DATE_TRUNC('month', MIN(order_date)) AS cohort_month
  FROM raw.orders
  GROUP BY user_id
),
-- Step 2: Calculate month intervals for repeat transactions
user_activities AS (
  SELECT 
    o.user_id,
    DATE_TRUNC('month', o.order_date) AS order_month,
    s.cohort_month,
    -- Month index: active month - first month
    DATE_DIFF('month', s.cohort_month, DATE_TRUNC('month', o.order_date)) AS period_month
  FROM raw.orders o
  JOIN user_signup s ON o.user_id = s.user_id
)
-- Step 3: Pivot count of returning users over relative periods
SELECT 
  cohort_month,
  COUNT(DISTINCT user_id) AS cohort_size,
  COUNT(DISTINCT CASE WHEN period_month = 0 THEN user_id END) AS month_0,
  COUNT(DISTINCT CASE WHEN period_month = 1 THEN user_id END) AS month_1,
  COUNT(DISTINCT CASE WHEN period_month = 2 THEN user_id END) AS month_2,
  COUNT(DISTINCT CASE WHEN period_month = 3 THEN user_id END) AS month_3
FROM user_activities
GROUP BY cohort_month
ORDER BY cohort_month;

5. A/B Test Evaluation & Readouts

A/B tests are evaluated using both SQL and Python notebooks, including power checks, randomization checks, and difference‑in‑proportions tests. Results are exported as tidy CSVs so Power BI can tell a clean, exec‑friendly story.

Pre‑test power analysis: Before launching an experiment, the pipeline computes required sample size for a minimum detectable effect (MDE). This ensures the test has enough power to detect meaningful lifts without running longer than necessary.

Randomization validation: After assignment, the pipeline checks balance on demographics and pre‑experiment behavior (e.g., signup date, device type, past purchase count). If balance is off, it flags the issue so teams can investigate assignment logic.

Primary analysis: Difference‑in‑proportions for conversion metrics (e.g., signup rate, purchase rate) with confidence intervals (delta method or bootstrap). The pipeline also checks for segment heterogeneity (e.g., does the variant work better on mobile vs desktop?) to inform targeting decisions.

Guardrail metrics: Retention and revenue per user are evaluated in the same experiment window to ensure the variant does not harm downstream metrics. If guardrails show degradation, the recommendation may be “hold” or “run longer” rather than “ship”.

Analysis, Insights & Business Impact

7. Observations & Key Findings

  • Funnel bottlenecks are visible and explainable.
    Observation: the largest synthetic drop‑off occurs between product_view and add_to_cart, with roughly 60% of sessions that view a product failing to add anything to cart. This pattern mirrors real e‑commerce sites where pricing, product availability, and checkout friction drive abandonment.
    Why it matters: identifying the bottleneck (product view → add‑to‑cart) directs experimentation and UX improvements to the highest‑impact area. Teams can A/B test pricing displays, add‑to‑cart buttons, or product recommendations without guessing where to focus.
  • Retention cohorts behave realistically.
    Observation: earlier cohorts (e.g., users who signed up in week 1) retain better than later cohorts, and retention curves flatten over time (week‑1 retention is higher than week‑4 retention, but the gap narrows). This matches typical e‑commerce behavior where early adopters are more engaged and retention stabilizes after the initial period.
    Why it matters: confirms that the synthetic environment is realistic enough to exercise cohort and retention logic credibly. Teams can use this pipeline to test retention strategies (e.g., email campaigns, loyalty programs) with confidence that the baseline behavior is plausible.
  • A/B tests become reproducible, not ad‑hoc.
    Observation: having generation, load, SQL flags, and stats in code means experiments can be re‑run and audited. When stakeholders ask “how did you compute the 4.2pp lift?”, the answer is “run this notebook” rather than “I used this tool and clicked these buttons”.
    Why it matters: reproducibility builds trust and enables debugging. If results look wrong, teams can trace back through the code to find the issue. This is closer to how mature product analytics teams treat experiments than one‑off analyses in spreadsheets or BI tools.
  • Device and channel segmentation reveal different behaviors.
    Observation: desktop users show higher checkout completion rates than mobile, and users from different acquisition channels (e.g., organic vs paid) have different retention profiles. These patterns are consistent with real e‑commerce data where device and channel correlate with intent and engagement.
    Why it matters: segmentation helps teams target improvements (e.g., optimize mobile checkout flow) and allocate marketing spend (e.g., invest more in channels with better retention).

8. Trade-offs, Risks & Failure Modes

The stack makes deliberate trade-offs: synthetic data may not reflect real user behavior; A/B tests in this version use simple difference‑in‑proportions and power checks, not sequential testing or multi‑armed bandits. Funnel definitions (e.g., 30‑minute session timeout) can bias metrics if they do not match how users actually behave. Mitigation: validate funnel and cohort logic on a sample of real data; extend experiment evaluation with more advanced methods when needed.

9. Results & Business Impact (Indicative)

For hiring managers and interviewers, the project demonstrates:

  • Ability to design an analytics data model for e‑commerce funnels and retention. The pipeline shows understanding of how events, sessions, and orders relate, and how to structure data so that common questions (funnel conversion, cohort retention) can be answered efficiently.
  • Confidence with SQL transformations and reproducible pipelines. The SQL scripts are clean, documented, and designed for reuse. This is the kind of work that scales across many metrics and experiments.
  • Understanding of funnel and retention metrics beyond “just a chart”. The project shows awareness of sessionization trade‑offs, cohort assignment logic, and how to interpret retention curves in business terms.
  • Familiarity with experiment evaluation and statistical tooling. Power analysis, randomization checks, and confidence intervals are implemented correctly, showing that the candidate can run rigorous experiments, not just descriptive analyses.

Operational impact: A stack like this shortens the path from raw events to “ship or hold” decisions. Instead of waiting days for ad‑hoc SQL or notebook analyses, product and growth teams can run experiments, get readouts, and make decisions in hours. The reproducible pipeline also reduces the risk of mis‑interpreting funnel lifts or retention changes, which can lead to wrong product decisions and wasted engineering effort.

10. Recommendations

  • Anchor funnels and cohorts in a single, documented data model so BI and notebooks tell the same story.
  • Run power and randomization checks before reading A/B results; avoid peeking at significance mid‑experiment without sequential methods.
  • Expose core metrics in a lightweight dashboard (e.g., Streamlit) so product managers can self‑serve without opening BI.

11. Limitations & What We'd Improve

Data is synthetic; production would connect to real event and order streams. Sessionization and funnel definitions are fixed (30‑minute timeout); we would make them configurable and validate against real user behavior. A/B analysis uses basic stats; we would add sequential testing or bandits for more advanced experimentation.
