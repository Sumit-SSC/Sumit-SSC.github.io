Project Overview

E-commerce Product Analytics is an end-to-end analytics system that turns simulated clickstream and order data into decision-ready metrics for funnels, cohort retention, and A/B testing. It uses DuckDB + SQL as the core analytics engine, Python for data generation and orchestration, and produces Power BI–ready exports plus notebooks for statistical validation.

Instead of a single notebook, the project is structured like a real analytics stack: raw data generation, database loading, SQL transformations, Jupyter analyses, and BI dashboards—all governed by a clear run sequence and documentation.

Business Context & Problem Framing

E-commerce product and growth teams ask three recurring questions:

  • Where are users dropping in the funnel? (visit → view → add-to-cart → checkout → purchase)
  • How well do cohorts retain? (by signup week, channel, device, or campaign)
  • Did this experiment actually work? (and is the effect size worth shipping?)

Many portfolio projects stop at EDA or a single model, but real teams need a reproducible analytics pipeline that:

  • Generates or ingests realistic clickstream + order data.
  • Materializes consistent, queryable tables for funnels and cohorts.
  • Supports A/B test readouts with statistically valid methods.
  • Exports tidy tables for BI tools so decisions can be communicated clearly.

Example: sessionization (30-minute timeout) and funnel flagging: 

-- Sessionization (simplified)
CREATE TABLE analytics.user_sessions AS
SELECT
  user_id,
  session_id,
  MIN(event_time) AS session_start,
  MAX(event_time) AS session_end,
  COUNT(*) AS event_count
FROM (
  SELECT
    user_id,
    event_time,
    SUM(session_break) OVER (
      PARTITION BY user_id
      ORDER BY event_time
    ) AS session_id
  FROM (
    SELECT
      user_id,
      event_time,
      CASE
        WHEN LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time)
             IS NULL THEN 1
        WHEN event_time - LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time)
             > INTERVAL '30' MINUTE THEN 1
        ELSE 0
      END AS session_break
    FROM raw.events
  )
)
GROUP BY user_id, session_id;
-- Funnel flags
CREATE TABLE analytics.funnel_sessions AS
SELECT
  s.user_id,
  s.session_id,
  MAX(CASE WHEN e.event_name = 'product_view'     THEN 1 ELSE 0 END) AS viewed,
  MAX(CASE WHEN e.event_name = 'add_to_cart'      THEN 1 ELSE 0 END) AS added_to_cart,
  MAX(CASE WHEN e.event_name = 'checkout_start'   THEN 1 ELSE 0 END) AS checkout_started,
  MAX(CASE WHEN e.event_name = 'purchase_complete' THEN 1 ELSE 0 END) AS purchased
FROM analytics.user_sessions s
JOIN raw.events e
  ON s.user_id = e.user_id
 AND e.event_time BETWEEN s.session_start AND s.session_end
GROUP BY s.user_id, s.session_id;

4. A/B Test Evaluation

A/B tests are evaluated using both SQL and Python notebooks. The README and notebooks implement:

  • Pre-test power calculations.
  • Randomization checks (balance across covariates).
  • Difference-in-proportions tests for conversion.
  • Bootstrap confidence intervals for robustness.
from statsmodels.stats.proportion import proportions_ztest

def conversion_lift(control_success, control_total, variant_success, variant_total):
    stat, pval = proportions_ztest(
        [variant_success, control_success],
        [variant_total, control_total],
        alternative="larger",
    )
    control_rate = control_success / control_total
    variant_rate = variant_success / variant_total
    lift = variant_rate - control_rate
    return {"lift": lift, "p_value": pval}

These results are then exported to data/powerbi/ab_test_summary.csv so the BI layer can tell a clean, exec-friendly story.

Insights, Trade-offs & Business Impact

5. Key Findings

  • Funnel bottlenecks are visible and explainable. The largest synthetic drop-off occurs between product_view and add_to_cart, highlighting merchandising, pricing, and UX opportunities.
  • Retention cohorts behave realistically. Early cohorts show stronger repeat behavior, then retention curves flatten, mirroring typical e-commerce lifecycle patterns.
  • A/B tests become reproducible, not ad-hoc. Having the full sequence in code—generation, load, SQL flags, stats—means the experiment can be re-run and audited.

6. Trade-offs & Design Decisions

  • DuckDB as analytics engine. DuckDB provides a local, SQL-first experience without needing a full warehouse; easy to ship in a portfolio.
  • SQL-first metrics, Python-second. Core metrics live in SQL views and tables, while notebooks focus on visualization and statistical checks.
  • Synthetic data, realistic patterns. As with other projects, synthetic data protects privacy while still exercising realistic analytics logic.
