Project Overview

The AI Governance Workbench is a multi-module Streamlit application that simulates how real AI governance teams work: dashboards for risk KPIs, a data visualizer for governance datasets, an AI-assisted document summarizer and flashcards engine, a prototype compliance audit workflow, and a Gemini + RAG chat assistant over uploaded policy documents. The goal is to turn abstract AI policy into a concrete, end-to-end decision system that a governance team could actually use.

Rather than building “just another LLM demo”, this project is framed as a governance control room: a single place where policy owners, data analysts, and risk officers can see AI deployment health, summarize documentation, audit gaps, and ask questions in natural language—with a clear link back to the underlying text and data.

The implementation is backed by Google Gemini for generative tasks, FAISS for retrieval, and a modular Streamlit layout that can be deployed on Streamlit Cloud or run locally via uv or pip.

Business Context & Problem Framing

Most organizations now experiment with AI and LLMs, but governance workflows are still spreadsheets + PDFs + emails. Policy documents live in shared drives, model cards are scattered, and risk owners rely on manual reviews and status meetings. That makes it hard to answer basic questions:

  • Which AI systems are in scope for a particular policy or regulation?
  • Where are the gaps in documentation and evidence?
  • How are governance KPIs trending: incidents, exceptions, overdue reviews?
  • Can a non-technical stakeholder ask, in plain language, “Are we compliant with X?” and see why?

Governance teams also need repeatable workflows: they perform the same steps (collect policies, review against frameworks, document evidence, brief stakeholders) across multiple models and products. A governance workbench should make those workflows faster, more auditable, and easier to explain.

Framing

Treat AI governance as a decision system: inputs are policies, model inventories, incidents, and controls; outputs are decisions about risk acceptance, mitigations, and go/no-go for deployments. The workbench is a prototype of that system.

Solution Architecture & Approach

1. Data & Event Model

The workbench organizes governance information into a small set of conceptual tables:

  • Systems: AI/ML systems in scope (id, owner, domain, data type, risk tier).
  • Policies & Controls: documents, sections, and mapped control requirements.
  • Evidence items: links to model cards, risk assessments, test reports, and sign-offs.
  • Incidents & exceptions: timestamps, severity, system, owner, resolution state.
  • Usage logs: user interactions in the app (what was summarized, which checks ran).

In the current version, these entities are simulated or loaded from CSVs, but the layout mirrors how a real governance team would want to track AI systems and their obligations.

2. Streamlit Modules

The application is organized as multiple Streamlit “views”, each solving a specific part of the governance workflow:

  1. Governance Dashboard – KPI tiles (systems in scope, open incidents, overdue reviews), time-series charts for incidents and exceptions, heatmaps by domain and risk tier.
  2. Data Visualizer – upload CSV/JSON/XLSX or use sample datasets; build bar/line/scatter/heatmap charts, with automatic summary stats for selected columns.
  3. Document Summarizer & Flashcards – upload policy PDFs/TXT/DOCX, run Gemini summarization, generate Q&A-style flashcards to quickly teach the policy to someone new.
  4. Compliance Audit Prototype – scan a policy document for the presence of predefined governance indicators (e.g., data lineage, human oversight, fairness) and generate a markdown report.
  5. AI Chat Assistant (with RAG) – ask questions like “Where do we mention human oversight?” or “What are our obligations under X?”, using FAISS-based retrieval over uploaded documents plus Gemini responses.

3. LLM & RAG Design

The LLM layer uses Google Gemini as the foundation model and a simple RAG pipeline for document-grounded answers: 

import google.generativeai as genai
from my_rag import embed_chunks, build_faiss_index, retrieve_top_k

genai.configure(api_key=os.environ["GOOGLE_API_KEY"])

def answer_with_rag(question: str, documents: list[str]) -> str:
    # 1) Chunk + embed
    chunks = chunk_documents(documents)
    index = build_faiss_index(embed_chunks(chunks))

    # 2) Retrieve relevant context
    context_chunks = retrieve_top_k(index, question, k=6)
    context_text = "\n\n".join(context_chunks)

    # 3) Ask Gemini with explicit grounding instructions
    prompt = f"""
    You are an AI governance assistant. Answer using ONLY the context below.
    If you are not sure, say you don't know.

    Context:
    {context_text}

    Question: {question}
    """

    response = genai.GenerativeModel("gemini-1.5-flash-latest").generate_content(prompt)
    return response.text.strip()

The key design choice is to treat the LLM as a decision-support layer, not a source of truth. The app always keeps a clear separation between:

  • Raw policy text (stored, chunked, and retrievable)
  • LLM summaries and answers, which are derived views of that text

4. Local Development & Deployment

The README documents two main ways to run the app: uv (recommended) and pip + venv. 

# Clone the repository
git clone https://github.com/Sumit-SC/MBA-AI_Governance_LLM-App.git
cd MBA-AI_Governance_LLM-App

# Option A — uv (recommended)
uv init --name "mba-ai-governance-llm-app"
uv add -r requirements.txt
uv venv
uv sync

# Configure secrets (.streamlit/secrets.toml)
echo 'GOOGLE_API_KEY="YOUR_GEMINI_API_KEY_HERE"' > .streamlit/secrets.toml

# Run the app
uv run streamlit run app.py

The same app is also deployable on Streamlit Community Cloud; the README includes a ready-to-use “Deploy to Streamlit” badge and guidance on storing the Gemini API key in Streamlit secrets.

Observations, Trade-offs & Impact

5. Observations & Key Findings

  • Governance can be productized. Turning static documents into interactive modules (dashboards, summarizer, audits) makes governance feel like a workflow, not a one-off review.
  • LLM value is highest in “last mile” tasks. Gemini and RAG work best when they summarize, prioritize, or explain; the app still keeps humans in the loop for judgement calls.
  • RAG needs good information design. Chunking, metadata (system, domain, control type), and simple filters dramatically improve answer quality compared to “just call the LLM”.
  • Governance KPIs are as important as model metrics. Tracking “incident aging”, “overdue reviews”, and “evidence coverage” is as critical as precision/recall in real-world AI programs.

6. Trade-offs & Risks

The workbench intentionally makes several trade-offs:

  • Synthetic vs real data. To avoid handling sensitive data, the app uses synthetic or sample datasets. This keeps it portfolio-safe, but real-world integration (e.g., into model registries or ticketing systems) would require additional engineering and security work.
  • Rule-based audits vs dynamic policies. The compliance audit prototype uses a fixed set of indicators (e.g., “mentions human oversight”). In production, organizations would want configurable rule sets and versioning per framework.
  • Model access and cost. Gemini API usage has cost and rate limits. The app uses gemini-1.5-flash and caches or constrains calls where possible, but a real deployment would need quotas, logging, and fallbacks.
  • Answer reliability. Even with RAG, LLMs can occasionally overstate or misinterpret policy text. The app mitigates this by surfacing the underlying chunks and treating outputs as decision support, not binding truth.

7. Business Impact & Decision Outcomes

While this project runs on synthetic and demo data, it mirrors impact levers that real AI governance leaders care about:

  • Reducing time-to-review for AI policies and model documentation.
  • Improving evidence coverage by mapping controls to actual artifacts, not just checklists.
  • Providing a central workspace where policy owners, AI teams, and risk officers collaborate.
  • Making go/no-go decisions on AI deployments more transparent and auditable.

For interviewing and portfolio purposes, the workbench demonstrates the ability to design full decision systems, not just isolated models or dashboards.

8. Limitations & Future Scope

The current version is a strong demo, but there are clear paths to production-grade maturity:

  • Integrate with model registry / inventory. Link systems and controls directly to a model registry (e.g., MLflow, internal inventory) instead of simulated CSVs.
  • Advanced policy parsing. Use NLP for entity extraction (e.g., references to data types, roles, legal articles) to enrich the search and audit views.
  • Multi-user workflows. Add authentication and persistent state so multiple reviewers can work on the same system and track who did what, when.
  • Configurable rule engine. Let governance teams define and version their own rule sets per framework (e.g., EU AI Act, NIST AI RMF).
