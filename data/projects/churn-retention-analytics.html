Project Overview

Churn & Retention Analytics is a full lifecycle project that takes synthetic subscription data from raw events to SQL features, ML risk scores, and a Streamlit retention simulator for ROI-based decision making. It combines analytics engineering, modeling, and product thinking into a single, interview-ready system.

The system is designed as if it were serving a SaaS or subscription business: it identifies at-risk customers in a 31–45 day window, prioritizes segments by risk and value, and helps test retention strategies (discounts, support upgrades, feature unlocks) under cost and ROI constraints.

Business Context & Problem Framing

Churn is a critical business problem for SaaS, telecom, fintech, and any subscription or recurring revenue model. Leaders ask:

  • Who is about to churn, and when?
  • Which customers are worth saving? (high value, high risk)
  • Which retention levers pay back? (discounts, support, features)

Many churn projects stop at a predictive model; this project goes further by:

  • Generating realistic lifecycle data for ~100k customers.
  • Building a SQL-first feature pipeline with clear data quality checks.
  • Training Logistic Regression and Random Forest models for risk scoring.
  • Surfacing these scores and features in a Streamlit retention simulator where business users can play with strategies and see ROI.

3. SQL Feature Engineering & Data Quality

The sql/ layer turns raw usage, billing, and support logs into analysis-ready features:

  • 30-day usage trends.
  • Payment issues and dunning status.
  • Support ticket volume and recency.
  • Lifecycle stage (Onboarding, Active, At-Risk, Churned).
WITH usage_trends AS (
  SELECT
    customer_id,
    AVG(usage_minutes) AS avg_usage_minutes,
    (
      AVG(CASE WHEN usage_date >= CURRENT_DATE - INTERVAL '30' DAY THEN usage_minutes END)
      - AVG(CASE WHEN usage_date < CURRENT_DATE - INTERVAL '30' DAY THEN usage_minutes END)
    ) / NULLIF(AVG(CASE WHEN usage_date < CURRENT_DATE - INTERVAL '30' DAY THEN usage_minutes END), 0)
    AS usage_trend_30d
  FROM staging.usage_daily
  GROUP BY customer_id
)
SELECT
  u.customer_id,
  u.avg_usage_minutes,
  u.usage_trend_30d,
  p.payment_issue_count_90d,
  s.support_ticket_count_90d
FROM usage_trends u
LEFT JOIN features.payment_signals p USING (customer_id)
LEFT JOIN features.support_signals s USING (customer_id);

  • Usage trend (30-day).
  • Number of payment issues.
  • Support ticket volume.
  • Plan tier.
  • Active days.

5. Retention Strategy Simulator (Streamlit)

The Streamlit retention simulator is where models and features become business decisions:

  • Filter by segment (plan, region, lifecycle stage).
  • Set risk thresholds and choose interventions (discount, support, feature unlock).
  • Estimate churn reduction per strategy.
  • Compute intervention cost and retained revenue → net ROI.

This is surfaced in the portfolio via the streamlit_url field when deployed, turning the project into a live, explorative tool rather than a static report.

Insights, Findings & Business Impact

6. Key Results (From README)

  • Baseline churn: ~7.58% across the synthetic portfolio.
  • Random Forest performance: ROC AUC 0.96, Precision@Top 10% = 51%.
  • Top 10% risk customers: 51% churn probability vs 7.58% baseline.
  • Best ROI strategy: Priority support on high-value customers, yielding ~2.3× ROI in the simulator.

7. Executive Insights

  • Usage decline is the strongest signal. Declining 30-day usage is a more powerful predictor than static attributes alone, supporting lifecycle-based interventions (e.g., reactivation campaigns).
  • Plan tier structures risk. Free/Basic tiers are much riskier than Enterprise, which supports tier-specific strategies and SLAs.
  • Payment and support friction compound churn. Payment issues and heavy support load together create a 4.5× higher risk segment in synthetic data.
  • Intervention timing matters. The 31–45 day window provides a meaningful chance to intervene before churn becomes irreversible.

8. Trade-offs, Limitations & Future Work

  • Synthetic data: safer for a portfolio, but real-world evaluation would refine thresholds and calibration.
  • Focus on tabular ML: deep learning is intentionally avoided; the system emphasizes features and explainability.
  • No production scheduler: pipeline is triggered manually via scripts; in production, this would be Airflow/DBT/Jobs.
  • Future scope: add cohort dashboards in a BI tool and integrate with real CRM or billing systems.
