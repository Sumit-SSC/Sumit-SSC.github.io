This case study walks through how I designed an e-commerce analytics workflow that covers sessionization, funnel conversion, cohort retention, and A/B test evaluation, all built on the principle that SQL is the source of truth and notebooks validate statistics.

The Business Problem: Product Teams Need Reliable Metrics

Product teams need answers to a small set of high-impact questions: where users drop in the funnel, which cohorts retain better over time, and whether an experiment genuinely improved outcomes. The challenge is that raw clickstream data is noisy: multiple devices per user, long gaps between events, partial funnels, bots, and outliers.

I've seen teams spend weeks building dashboards only to realize later that their metrics are inconsistent—session counts don't match between tools, retention curves look wrong, and experiment results can't be reproduced. This project focuses on building "analysis-grade" tables that keep metrics consistent and reproducible.

The key insight is that analytics engineering—building curated, documented tables from raw data—is just as important as the analysis itself. If your sessionization logic is inconsistent, your funnel metrics will be wrong. If your cohort definitions drift, your retention curves will be misleading. This project demonstrates how to build tables that are stable, documented, and validated.

The Core Challenge

How do you turn noisy clickstream data into reliable, reproducible metrics that product teams can trust? The answer is SQL-first transformations with clear sessionization logic, consistent funnel definitions, and statistical validation in notebooks.

Data Pipeline Design: From Raw Events to BI-Ready Tables

The pipeline follows a logical flow: raw events → sessionization → funnel flags → cohort tables → BI exports. Each stage is SQL-based and documented, making it reproducible and testable. 

Raw Data Layer
  ├─ events_raw (user_id, event_type, timestamp, properties)
  ├─ orders_raw (order_id, user_id, amount, timestamp)
  └─ users_raw (user_id, signup_date, attributes)

Sessionization Layer (SQL)
  ├─ sessions (session_id, user_id, start_time, end_time, device, channel)
  └─ session_events (session_id, event_sequence, conversion_flag)

Funnel Layer (SQL)
  ├─ funnel_sessions (session_id, step_1_view, step_2_cart, step_3_checkout, step_4_purchase)
  └─ funnel_users (user_id, first_purchase_date, lifetime_value)

Cohort Layer (SQL)
  ├─ cohorts (cohort_month, user_id, first_session_date)
  └─ retention (cohort_month, period, retained_users, retention_rate)

BI Exports (Power BI-ready)
  ├─ bi_sessions_daily (date, sessions, conversions, conversion_rate)
  ├─ bi_funnel_steps (step, sessions, dropoff_rate)
  └─ bi_cohort_retention (cohort_month, period_0, period_1, ..., period_12)

Methodology: Building Reliable Analytics Tables

1. Sessionization: Defining User Sessions

The first challenge is turning a stream of events into sessions. I use a time-window approach: if a user has no activity for 30 minutes, that's a session boundary. Within a session, I track:

  • Session start and end time
  • Device type (mobile/desktop/tablet)
  • Channel (organic/search/social/direct)
  • Conversion flag (did this session result in a purchase?)
  • Funnel steps completed (view → cart → checkout → purchase)

Sessionization SQL Example

WITH event_gaps AS (
  SELECT 
    user_id,
    event_timestamp,
    LAG(event_timestamp) OVER (PARTITION BY user_id ORDER BY event_timestamp) AS prev_timestamp,
    EXTRACT(EPOCH FROM (event_timestamp - LAG(event_timestamp) OVER 
           (PARTITION BY user_id ORDER BY event_timestamp))) / 60 AS minutes_since_last_event
  FROM staging.events
),
session_starts AS (
  SELECT 
    user_id,
    event_timestamp,
    CASE 
      WHEN minutes_since_last_event > 30 OR minutes_since_last_event IS NULL THEN 1
      ELSE 0
    END AS is_session_start
  FROM event_gaps
),
sessions AS (
  SELECT 
    user_id,
    event_timestamp AS session_start,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY event_timestamp) AS session_num
  FROM session_starts
  WHERE is_session_start = 1
)
SELECT 
  s.user_id,
  s.session_start,
  LEAD(s.session_start) OVER (PARTITION BY s.user_id ORDER BY s.session_start) AS session_end,
  COUNT(DISTINCT e.event_type) AS event_types_count
FROM sessions s
LEFT JOIN staging.events e ON e.user_id = s.user_id 
  AND e.event_timestamp >= s.session_start 
  AND (e.event_timestamp < LEAD(s.session_start) OVER (PARTITION BY s.user_id ORDER BY s.session_start) 
       OR LEAD(s.session_start) OVER (PARTITION BY s.user_id ORDER BY s.session_start) IS NULL)
GROUP BY s.user_id, s.session_start

2. Funnel Analysis: Tracking Conversion Steps

Once sessions are defined, I compute funnel flags for each step: product view, add to cart, checkout start, purchase. The key is tracking both session-level and user-level funnels, since users can complete funnels across multiple sessions.

The funnel table includes:

  • Step completion flags (boolean for each step)
  • Time-to-step metrics (how long from session start to each step)
  • Dropoff rates (percentage of users who drop at each step)
  • Device/channel breakdowns (funnel performance by segment)

3. Cohort Retention: Measuring Long-Term Value

Cohort analysis tracks how groups of users behave over time. I define cohorts by first session date (or first purchase date), then compute retention rates for each period (day 0, day 7, day 30, etc.). The key is consistent cohort definitions and date alignment—all users in a cohort should be measured from the same starting point.

Cohort Retention SQL Example

WITH first_sessions AS (
  SELECT 
    user_id,
    MIN(session_start) AS first_session_date,
    DATE_TRUNC('month', MIN(session_start)) AS cohort_month
  FROM sessions
  GROUP BY user_id
),
user_activity AS (
  SELECT 
    fs.user_id,
    fs.cohort_month,
    s.session_start,
    DATE_TRUNC('month', s.session_start) AS activity_month,
    (DATE_PART('year', s.session_start) - DATE_PART('year', fs.first_session_date)) * 12 +
     (DATE_PART('month', s.session_start) - DATE_PART('month', fs.first_session_date)) AS period
  FROM first_sessions fs
  JOIN sessions s ON s.user_id = fs.user_id
),
cohort_sizes AS (
  SELECT 
    cohort_month,
    COUNT(DISTINCT user_id) AS cohort_size
  FROM first_sessions
  GROUP BY cohort_month
),
retention AS (
  SELECT 
    cohort_month,
    period,
    COUNT(DISTINCT user_id) AS retained_users
  FROM user_activity
  GROUP BY cohort_month, period
)
SELECT 
  r.cohort_month,
  cs.cohort_size,
  r.period,
  r.retained_users,
  r.retained_users::FLOAT / cs.cohort_size AS retention_rate
FROM retention r
JOIN cohort_sizes cs ON cs.cohort_month = r.cohort_month
ORDER BY r.cohort_month, r.period

4. A/B Test Evaluation: Statistical Rigor

When evaluating experiments, I use difference-in-proportions tests (z-tests) for conversion rates and bootstrap confidence intervals for more complex metrics. The key is checking randomization (are control and treatment groups balanced?), computing lift with confidence intervals, and providing clear ship/hold recommendations.

The notebook includes:

  • Pre-test power calculations (sample size needed to detect meaningful lift)
  • Randomization checks (demographic balance, baseline metrics)
  • Difference-in-proportions test (z-test for conversion lift)
  • Bootstrap confidence intervals (for retention/revenue metrics)
  • Business readout (ship/hold recommendation with monitoring plan)

Key Insights: What the Data Revealed

1. Funnel Bottleneck: Product View → Add to Cart

The largest drop-off occurs between product view and add to cart. Only 35% of users who view a product add it to cart, compared to 60% who add to cart then proceed to checkout. This suggests the product page might need optimization—better product images, clearer pricing, or social proof (reviews, ratings).

2. Device Behavior: Desktop Outperforms Mobile

Desktop shows higher checkout completion rates (45% vs. 32% on mobile), likely due to payment friction on mobile devices. Mobile users are more likely to abandon at checkout, suggesting the mobile checkout flow needs improvement—perhaps one-click payment options or saved payment methods.

3. Cohort Retention: Early Cohorts Retain Better

Early cohorts (users who signed up in the first few months) show better retention than later cohorts. Month 1 cohort has 40% retention at month 6, while Month 6 cohort has only 25% retention at month 6. This suggests stronger onboarding effects for early users, or that the product-market fit was stronger in the early days.

4. Experiment Rigor: Statistical Validation Matters

Z-tests and bootstrap confidence intervals help avoid "false wins" in noisy funnels. In one experiment, the variant showed a 2% conversion lift, but the 95% confidence interval included zero, so we held the change. In another, a 4.2% lift with a tight CI (1.1% to 7.3%) gave us confidence to ship.

How This Maps to a Real Analytics Stack

The structure mirrors analytics engineering best practices used at companies like dbt Labs, where SQL transformations are version-controlled, tested, and documented. The key principles are:

  • Data contracts: Stable grains for BI (session/day/cohort). Once a table is defined, its grain shouldn't change without versioning. This prevents BI dashboards from breaking when underlying logic changes.
  • Reproducibility: Deterministic SQL transformations + validation notebooks. Anyone should be able to run the pipeline and get the same results. This requires careful handling of randomness (seeds) and time-based logic (consistent date functions).
  • Decision support: Experiment readouts written for product managers (ship/hold + monitoring plan). The analysis should answer "what should we do?" not just "what happened?"
  • Documentation: Every table has a clear purpose, grain, and definition. This makes it easier for new team members to understand the data model and prevents metric drift.

Technical Deep Dive: Sessionization Logic

Sessionization is the foundation of e-commerce analytics. Get it wrong, and all downstream metrics will be wrong. Here's how I approach it:

Time-Window Sessionization

I use a 30-minute inactivity window: if a user has no events for 30 minutes, that's a session boundary. This is a common industry standard, but it's configurable. Some teams use 15 minutes for mobile apps, 60 minutes for web.

User Stitching

Users can have multiple devices and anonymous sessions before signing up. I use a combination of user_id (when available) and device_id (for anonymous sessions) to stitch sessions together. This requires careful handling of the transition from anonymous to authenticated.

Session Attributes

Each session gets attributes that are useful for analysis:

  • Device type (mobile/desktop/tablet) from user agent parsing
  • Channel (organic/search/social/direct) from referrer analysis
  • Conversion flag (did this session result in a purchase?)
  • Session duration (time from first to last event)
  • Event count (how many events in this session)

A/B Testing: Statistical Validation in Practice

When evaluating experiments, I follow a rigorous process:

1. Pre-Test Power Analysis

Before running an experiment, I calculate the sample size needed to detect a meaningful lift. For a 5% conversion rate, detecting a 10% relative lift (0.5 percentage points) with 80% power requires ~15,000 users per variant. This helps set experiment duration and avoid underpowered tests.

2. Randomization Checks

I check that control and treatment groups are balanced on demographics, device types, and baseline metrics. If groups aren't balanced, the experiment might be invalid. Common checks include chi-square tests for categorical variables and t-tests for continuous variables.

3. Difference-in-Proportions Test

For conversion rates, I use a z-test to compare proportions between variants. The test statistic is:

z = (p₁ - p₀) / √(p(1-p)(1/n₁ + 1/n₀))

where p is the pooled proportion, p₁ is treatment conversion rate, p₀ is control conversion rate

4. Bootstrap Confidence Intervals

For metrics like revenue per user or retention, I use bootstrap confidence intervals. This involves resampling the data thousands of times and computing the metric for each sample, then taking the 2.5th and 97.5th percentiles as the confidence interval. This is more robust than assuming normality.

Challenges & Lessons Learned

Challenge 1: Sessionization Edge Cases

Edge cases include users with multiple tabs open (events arrive out of order), bots/crawlers (generate fake sessions), and timezone issues (events from different timezones). The solution is careful event ordering, bot filtering, and consistent timezone handling.

Challenge 2: Cohort Date Alignment

Retention curves can look wrong if cohort dates aren't aligned properly. Users who sign up on Jan 31 should be measured from Jan 31, not Feb 1. The solution is consistent date truncation (always use the same function) and clear documentation of cohort definitions.

Challenge 3: Statistical Significance vs. Business Significance

A statistically significant lift might not be business-significant. A 0.1% conversion lift with p < 0.05 might not be worth the engineering effort to ship. The solution is to frame results in business terms: "This change increases revenue by $X per month" rather than just "p < 0.05".

What I'd Build Next: Advanced Analytics

1. Multi-Touch Attribution

Build attribution tables that assign credit for conversions across multiple touchpoints. This would include first-touch, last-touch, linear, and data-driven attribution models. The tables would help marketing teams understand which channels drive conversions.

2. Incrementality Testing

Design holdout experiments to measure the incremental impact of marketing campaigns on retention and revenue, not just conversion. This requires careful experimental design to avoid selection bias and ensure valid causal inference.

3. Anomaly Detection

Build automated anomaly detection on funnel steps and event volumes. This would alert teams when conversion rates drop unexpectedly or when traffic spikes occur, helping catch issues before they impact business metrics.

4. Semantic Layer

Create centralized metric definitions (e.g., "active user" = user with session in last 30 days) to reduce BI drift. This would use a tool like dbt's metrics layer or a custom semantic layer that ensures consistent definitions across dashboards and reports.
