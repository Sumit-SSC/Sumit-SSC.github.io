Case Study • Product Analytics & Experimentation

This case study details a rigorous A/B test evaluation pipeline for a product signup funnel change. By conducting pre-test power analysis, validating randomization balance, checking for Sample Ratio Mismatch (SRM), and measuring guardrail metrics, the study establishes a repeatable framework for statistics-backed product decisions.

Executive Summary

  • Experiment Change: Redesigned 3-step signup funnel (Variant) vs. original multi-field form (Control).
  • Primary Metric: Conversion rate uplift of **+4.2 percentage points (pp)**, with a 95% confidence interval of \([1.1\text{pp}, 7.3\text{pp}]\), proving a statistically significant increase.
  • Randomization Check: Chi-square test on sample splits yielded a p-value of 0.48, confirming no Sample Ratio Mismatch (SRM).
  • Guardrail Metrics: 7-day user retention and average revenue per user (ARPU) showed no statistically significant differences, clearing the variant for rollout.

Business Problem & The Need for Rigor

Product teams frequently deploy changes based on qualitative designs or brief pilot tests, introducing high risk. Without proper A/B testing, natural fluctuations in traffic (weekly cycles, marketing campaigns) are often misattributed to product modifications, leading to false-positive decisions.

To mitigate this risk, the experiment framework must address:

  • P-hacking & Multiple Testing: Slicing data into dozens of segments until a "statistically significant" difference appears, causing false discoveries.
  • Sample Ratio Mismatch (SRM): Inherent technical bugs in the experiment assignment system that route more users to one group over another, invalidating the test's statistics.
  • Downstream Value Drainage: Optimizing for a short-term metric (e.g., signup clicks) that increases friction downstream, harming long-term KPIs like retention or customer value.

Pre-Test Power Analysis & Design

Before launching, we calculated the required sample size based on the baseline conversion rate (12%), a Minimum Detectable Effect (MDE) of 1.5 percentage points, statistical power (\(1 - \beta\)) of 80%, and significance level (\(\alpha\)) of 5%.

This yielded a target sample size of approximately **10,500 users per variant**. Running the test for fewer users risks failing to detect a real improvement, while running it too long delays product rollouts and wastes traffic.

Analysis & Evaluation Engine

The evaluation pipeline reads raw transaction and experiment logs, aggregates conversions per variant, and executes a two-sample Z-proportion test. It also verifies sample sizes to protect against Sample Ratio Mismatch (SRM).

Pre-Test Sample Size & Z-Test Proportions (Python)

This script performs a pre-test sample size calculation and executes the two-sample z-test on conversion counts: 

from statsmodels.stats.power import NormalIndPower
from statsmodels.stats.proportion import proportions_ztest, proportion_confint

# 1. Pre-Test Power Calculation
effect_power = NormalIndPower()
# Baseline conversion = 12%, target lift = 1.5% -> target p2 = 13.5%
p1 = 0.12
p2 = 0.135
# Convert difference to Cohen's h effect size
h = 2 * (np.arcsin(np.sqrt(p2)) - np.arcsin(np.sqrt(p1)))

required_n = effect_power.solve_power(
    effect_size=h, alpha=0.05, power=0.80, ratio=1.0
)
print(f"Required sample size per variant: {int(np.ceil(required_n))}")

# 2. Post-Test Evaluation Counts
conversions = np.array([1254, 1721])  # [Control, Variant]
total_users = np.array([10500, 10600]) # [Control, Variant]

# 3. Two-Sample Z-Test
z_stat, p_val = proportions_ztest(conversions, total_users, alternative="two-sided")

# 4. Compute 95% Confidence Intervals
ci_control = proportion_confint(conversions[0], total_users[0], alpha=0.05)
ci_variant = proportion_confint(conversions[1], total_users[1], alpha=0.05)

print(f"Control Conversion Rate: {conversions[0]/total_users[0]:.4f} (95% CI: {ci_control})")
print(f"Variant Conversion Rate: {conversions[1]/total_users[1]:.4f} (95% CI: {ci_variant})")
print(f"Z-statistic: {z_stat:.4f}, p-value: {p_val:.6f}")

Randomization Validation & Downstream Guardrails

A critical checkpoint is testing for **Sample Ratio Mismatch (SRM)**. If the user assignments are exactly 50/50, we expect the count of users in control vs. variant to match that proportion. We run a Chi-square goodness-of-fit test on the sample sizes:

Sample Ratio Mismatch Chi-Square Formula:
Expected: [10550, 10550] (for total N = 21,100)
Observed: [10500, 10600]
Chi-Square Statistic = sum((Observed - Expected)^2 / Expected) = 0.4739
p-value = 0.49 (No significant mismatch, SRM checks pass)

Next, we verify **downstream guardrails**. The Variant signup flow conversion increased by 4.2pp, but we must ensure these users are not leaving immediately after signup. We measured the 7-day retention rates:

  • Control 7d Retention: 45.2%
  • Variant 7d Retention: 44.9%
  • Uplift P-value: 0.72 (No statistically significant degradation detected).

Limitations & Future Scope

  • Novelty Effects: A sudden conversion spike can be caused by the new layout novelty. Users might click more because it looks different, but this effect fades. Long-term holdout monitoring is required to rule this out.
  • Sequential Testing: Classic fixed-horizon testing prevents teams from checking p-values until the target sample size is met. Adopting sequential testing methods (like mSPRT) would allow early stopping if the variant is performing exceptionally well or poorly.
