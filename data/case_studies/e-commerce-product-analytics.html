This is written like a Product Analytics / Analytics Engineering case study: a reproducible SQL-first pipeline for funnel, sessionization, cohort retention, and experiment readouts that can feed BI and leadership decisions.

Executive summary
  • Dataset: ~120k users, ~1M events, ~45k orders, ~400k sessions.
  • Outputs: session tables, funnel tables, cohort retention tables, A/B evaluation notebook, BI exports.
  • Point of view: SQL is the source of truth; notebooks validate statistics and guard against misleading conclusions.

1) Business problem

Product teams need answers to a small set of high-impact questions: where users drop in the funnel, what cohorts retain, and whether an experiment genuinely improved outcomes.

The challenge is that raw clickstream is noisy: multiple devices, long gaps between events, partial funnels, and bots/outliers. This project focuses on building “analysis-grade” tables that keep metrics consistent.

2) Data pipeline design

raw events + orders
  → sessionization (time-window + user stitching)
  → funnel flags (view → add_to_cart → checkout → purchase)
  → cohort tables (first purchase / first session cohorts)
  → BI exports (stable grain, documented definitions)

3) Methodology

  • Sessionization: define session boundaries and compute per-session attributes (device, channel, conversion).
  • Funnel: compute step completion and time-to-step metrics at user and session grain.
  • Cohorts: compute retention curves with clear cohort definitions and date alignment.
  • Experiment readout: difference-in-proportions tests + bootstrap confidence intervals (when appropriate) to validate lift.

4) Key insights (from the project)

  • Funnel bottleneck: biggest drop is typically between product view and add-to-cart.
  • Device behavior: desktop can outperform mobile on checkout completion (often due to payment friction).
  • Cohorts: early cohorts can retain better (onboarding + product-market fit effects).
  • Experiment rigor: z-tests + bootstrap CIs help avoid “false wins” in noisy funnels.

5) How this maps to a real analytics stack

The structure mirrors analytics engineering best practices: documented models, consistent metric definitions, and BI datasets built from curated tables—not ad hoc joins in dashboards.

  • Data contracts: stable grains for BI (session/day/cohort).
  • Reproducibility: deterministic SQL transformations + validation notebooks.
  • Decision support: experiment readouts written for product managers (ship/hold + monitoring plan).

6) What I’d build next (advanced)

  • Attribution: multi-touch attribution tables and channel quality metrics.
  • Incrementality: holdout designs for retention/revenue impact (beyond conversion lift).
  • Guardrails: automated anomaly detection on funnel steps and event volumes.
  • Semantic layer: centralized metric definitions to reduce BI drift.
