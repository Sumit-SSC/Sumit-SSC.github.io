Fraud systems that block every suspicious transaction minimize loss but maximize false positives—legitimate users get declined, support load spikes, and trust drops. This case study examines how to design fraud analytics that balance precision (catching fraud) with customer friction (false declines, step-up challenges)—using Deloitte and Stripe fraud prevention frameworks.

Executive Summary

  • Precision vs friction: Lowering the fraud block threshold reduces loss but increases false positives and customer friction; the optimum is a function of loss cost vs lifetime value at risk.
  • Fraud lifecycle: From first touch (account creation, login) through transaction and post-purchase; signals and rules should align to each stage.
  • Label delays: Fraud labels arrive late (chargebacks, disputes); use proxy labels and semi-supervised methods for real-time scoring while monitoring label lag.
  • Rules + ML layering: Rules for hard blocks (e.g., known bad); ML for rank-ordering; human review for gray zone; step-up (e.g., 3DS) to reduce friction for borderline cases.
  • Ops KPIs: Review queue depth, time-to-decision, overturn rate, and analyst accuracy to keep the human loop effective.

Business Context & Problem Framing

The False Decline Problem

Merchants and payment platforms face a fundamental tension: minimize fraud loss while minimizing friction for legitimate users. According to Deloitte's fraud analytics research, false declines—legitimate transactions incorrectly blocked as fraud—can cost merchants more in lost revenue than actual fraud losses. The mechanism: a declined customer may abandon the purchase entirely, costing not just the immediate transaction but potentially lifetime value.

Stripe's fraud prevention guides describe how leading platforms use layered approaches: rules for hard blocks, ML for risk scoring, step-up authentication (3DS) for borderline cases, and human review for gray zones. The key insight is that different risk levels require different interventions, not a binary accept/decline decision.

The Business Stakes

For e-commerce platforms, fraud rates typically range from 0.5-2% of transaction volume, while false decline rates can be 5-15% if systems are overly conservative. According to Gartner's fraud detection research, a 1% reduction in false decline rate can increase revenue by $50-200K annually for mid-sized merchants, often exceeding the value of preventing additional fraud. The optimal operating point balances these costs.

Label Delay and Proxy Labels

A critical challenge in fraud analytics is label delay: confirmed fraud labels arrive weeks or months after transactions (chargeback windows are typically 90-120 days). According to Stripe's fraud prevention research, this delay requires using proxy labels for real-time model training:

  • Proxy fraud labels: Transactions declined by rules or high-confidence ML scores (treated as likely fraud)
  • Proxy legit labels: Transactions accepted and not disputed within a shorter window (e.g., 30 days)
  • Delayed model retraining: Models retrain periodically (weekly/monthly) as confirmed labels arrive
-- Label Assignment Logic
WITH transaction_labels AS (
  SELECT 
    transaction_id,
    transaction_date,
    decision,
    risk_score,
    -- Proxy label: decline = likely fraud
    CASE WHEN decision = 'decline' THEN 'fraud' ELSE NULL END AS proxy_fraud_label,
    -- Confirmed label: chargeback = confirmed fraud
    CASE WHEN chargeback_filed = TRUE THEN 'fraud' 
         WHEN transaction_date < CURRENT_DATE - 120 AND chargeback_filed = FALSE THEN 'legit'
         ELSE NULL END AS confirmed_label
  FROM transactions
)
SELECT 
  transaction_id,
  COALESCE(confirmed_label, proxy_fraud_label) AS label,
  CASE WHEN confirmed_label IS NOT NULL THEN 'confirmed' ELSE 'proxy' END AS label_type
FROM transaction_labels;

Precision-Recall Trade-off

The fundamental trade-off in fraud detection is between precision (of blocks that are fraud) and recall (of fraud that is blocked). According to Gartner's fraud detection research, lowering the block threshold improves recall (catches more fraud) but degrades precision (more false positives). The optimal point depends on the relative cost of fraud vs. false declines.

Ops KPIs for Human Review

When transactions fall into the gray zone (moderate risk scores), human review is essential. Key operational metrics include:

  • Queue depth: Number of transactions awaiting review (target: < 100)
  • Time-to-decision: Average time from queue entry to decision (target: < 2 hours)
  • Overturn rate: % of ML declines that reviewers approve (indicates threshold calibration)
  • Analyst accuracy: % of reviewer decisions that align with eventual chargeback outcomes

Analytical / Experimental Approach

A/B Testing Threshold and Model Changes

The most direct way to optimize fraud systems is A/B testing threshold or model changes. However, fraud experiments require careful design because:

  • Label delay: Fraud outcomes take weeks/months to materialize, requiring long experiment durations
  • Holdout groups: A subset of transactions must be held out from fraud systems to measure true fraud rates (unbiased by intervention)
  • Revenue impact: Changes affect both fraud loss and legitimate transaction volume

According to Stripe's fraud prevention research, experiments should run for at least 4-6 weeks to capture full chargeback windows, with holdout groups representing 1-5% of traffic to measure baseline fraud rates.

Segment-Specific Threshold Optimization

Different customer segments have different fraud risk profiles and lifetime values, justifying segment-specific thresholds. According to Gartner's fraud detection research, common segmentation includes:

  • New vs returning customers: New customers have less behavioral history, requiring more conservative thresholds
  • High-value vs low-value: High-value customers justify more review resources and lower thresholds
  • Geographic segments: Different regions have different fraud patterns and chargeback rates
  • Payment method: Credit cards vs digital wallets have different risk profiles

Vintage Analysis for Model Decay

Fraud models decay over time as fraudsters adapt and patterns change. Vintage analysis—tracking fraud rates by transaction cohort over time—identifies when models need retraining. According to Deloitte's fraud analytics research, models typically need retraining every 3-6 months, with more frequent updates during periods of high fraud activity or new attack patterns. 

-- Vintage Analysis: Fraud Rate by Transaction Cohort
WITH transaction_cohorts AS (
  SELECT 
    DATE_TRUNC(transaction_date, WEEK) AS cohort_week,
    COUNT(*) AS total_transactions,
    SUM(CASE WHEN chargeback_filed = TRUE THEN 1 ELSE 0 END) AS fraud_count
  FROM transactions
  WHERE transaction_date >= CURRENT_DATE - 180
  GROUP BY cohort_week
)
SELECT 
  cohort_week,
  total_transactions,
  fraud_count,
  ROUND(100.0 * fraud_count / total_transactions, 2) AS fraud_rate_pct,
  -- Compare to baseline (first 4 weeks)
  ROUND(100.0 * fraud_count / total_transactions - 
        AVG(CASE WHEN cohort_week <= DATE_TRUNC(CURRENT_DATE - 120, WEEK) 
            THEN 100.0 * fraud_count / total_transactions END) OVER (), 2) AS fraud_rate_delta
FROM transaction_cohorts
ORDER BY cohort_week DESC;

  • Ensemble methods: Combine multiple models (e.g., gradient boosting + neural network) for better accuracy

Layered Decision Architecture

Effective fraud systems use layered architectures where different risk levels trigger different interventions:

  • Rules engine (hard blocks): Known bad actors, blacklisted devices/IPs, impossible travel patterns
  • ML scoring (rank-ordering): Risk scores for all transactions, used to prioritize review queue
  • Step-up authentication (3DS): For borderline cases, require additional authentication rather than declining
  • Human review (gray zone): Moderate risk scores (e.g., 0.6-0.8) go to review queue for analyst decision

According to Deloitte's fraud analytics research, layered approaches achieve 15-25% better total cost than single-layer systems because they reduce false declines without materially increasing fraud.

LLM-Assisted Case Summarization

Large language models enable automated summarization of transaction context for human reviewers. Rather than analysts manually reviewing raw data, LLMs can:

  • Generate human-readable summaries of transaction features and risk signals
  • Highlight anomalies (e.g., "First transaction from new device in different country")
  • Suggest decision rationale based on historical patterns
  • Surface similar historical cases for context

This follows HBR's framework for AI augmentation: automate routine analysis, escalate complex cases to human judgment.

Observations & Key Findings

Finding 1: Layered Approaches Outperform Single-Layer Systems

Across multiple platforms and industries, layered fraud systems (rules + ML + step-up + review) consistently achieve 15-25% lower total cost than single-layer systems. According to Deloitte's fraud analytics research, the mechanism is precision: rules catch obvious fraud, ML ranks moderate risk, step-up reduces false declines, and review handles edge cases. Each layer optimizes for different risk levels.

Finding 2: Segment-Specific Thresholds Improve Total Cost

Platforms that use segment-specific thresholds (e.g., lower threshold for new customers, higher for high-value customers) achieve 10-20% better total cost than uniform thresholds. The mechanism: different segments have different fraud rates and lifetime values, justifying different risk tolerances. According to Gartner's fraud detection research, new customers have 3-5x higher fraud rates than returning customers, justifying more conservative thresholds.

Finding 3: Step-Up Authentication Reduces False Declines Without Increasing Fraud

Step-up authentication (3DS, SMS verification) for borderline risk scores reduces false decline rates by 30-50% without materially increasing fraud rates. According to Stripe's fraud prevention research, the mechanism is friction: legitimate customers complete authentication, while fraudsters abandon. This enables lower thresholds (more borderline cases) without accepting more fraud.

Finding 4: Label Delay Must Be Accounted For

Fraud labels arrive 90-120 days after transactions (chargeback windows), requiring proxy labels for real-time model training. However, proxy labels have noise: some declined transactions are legitimate, some accepted transactions become fraud. Vintage analysis shows that models trained only on proxy labels can have 10-20% accuracy degradation compared to models trained on confirmed labels. The solution is periodic retraining as confirmed labels arrive.

Trade-offs, Risks & Failure Modes

Trade-off 1: Precision vs Recall

Lowering the fraud block threshold improves recall (catches more fraud) but degrades precision (more false positives). The optimal point depends on the relative cost of fraud vs false declines. For high-value merchants with low fraud tolerance, precision matters more; for high-volume merchants with thin margins, recall matters more.

Trade-off 2: Automation vs Human Review

More automation (higher thresholds, less review) reduces operational costs but increases false declines. More human review improves accuracy but increases costs and decision latency. The optimal balance depends on review capacity and cost per review.

Failure Mode: Adversarial Adaptation

Fraudsters adapt to fraud systems, learning thresholds and patterns. A system that blocks all transactions from certain IP ranges will see fraudsters switch IPs. A system that relies heavily on device fingerprinting will see fraudsters use different devices. The mitigation is continuous monitoring, periodic threshold review, and diversity of signals (don't rely on a single feature).

Failure Mode: Ignoring Label Delay

Models trained only on proxy labels (declines) can learn to decline everything, because declines are treated as fraud. This creates a feedback loop where the model becomes overly conservative. The fix is to use holdout groups (transactions not subject to fraud systems) to measure true fraud rates and calibrate models accordingly.

Decision Framework Applied

The decision framework that emerges from this analysis:

  1. Optimize for total cost: Minimize fraud loss + friction cost, not fraud rate alone
  2. Use layered architecture: Rules for hard blocks, ML for scoring, step-up for borderline, review for gray zone
  3. Set segment-specific thresholds: Different risk tolerances for new vs returning, high-value vs low-value customers
  4. Account for label delay: Use proxy labels for real-time training, confirmed labels for periodic retraining
  5. Monitor ops KPIs: Track queue depth, time-to-decision, overturn rate, and analyst accuracy
  6. Continuous adaptation: Monitor for adversarial patterns and refresh models every 3-6 months

Limitations & What We'd Improve

Limitations of This Analysis

  • Label delay and proxy quality: Proxy labels (declines) are noisy indicators of fraud; confirmed labels arrive too late for real-time model updates
  • Cold-start for new users/merchants: New users have limited behavioral history, making risk assessment difficult; new merchants have no historical fraud patterns
  • Adversarial adaptation: Fraudsters continuously adapt to fraud systems, requiring constant monitoring and model updates
  • Regulatory constraints: PSD2, strong customer authentication, and other regulations constrain fraud system design

What We'd Improve

  • Better proxy labels: Use multiple proxy signals (decline, step-up completion, review outcome) to improve label quality
  • Causal evaluation of step-up impact: Use causal inference methods to measure true impact of step-up authentication on fraud and friction
  • Adaptive thresholds: ML models that learn optimal thresholds by segment and adapt to changing fraud patterns
  • Real-time model updates: Online learning systems that update models incrementally as new labels arrive, reducing label delay impact
